# supertest Skill — Design Spec

**Date:** 2026-04-27  
**Status:** Draft  
**Author:** hojinzs

---

## Overview

`supertest`는 QA 전문가처럼 테스트 계획을 수립하고, 자동으로 테스트를 실행하며, HTML 형식으로 보고서를 생성하는 Claude Code 스킬이다. 코드 리뷰 기능은 없으며, 테스트 실행과 결과 보고에 집중한다.

---

## User Scenario

1. 사용자가 `supertest` 스킬을 실행
2. 스킬이 SUPERTEST.md 존재 여부 확인 (없으면 init 또는 자동 감지)
3. PR 또는 사용자 입력으로 테스트 대상 기능 파악
4. TC 작성 (GREEN CASE + EDGE CASE)
5. TC 기준으로 테스트 자동 실행
6. HTML 보고서 3종 생성, PR 코멘트 게시 (설정 시)

---

## Phase Flow

### Phase 0: SUPERTEST.md 확인

```
SUPERTEST.md 존재?
├─ 있음 → 파일 읽기 → Phase 1
├─ 없음 + 대화형 환경 → init 인터뷰 → SUPERTEST.md 생성 → Phase 1
└─ 없음 + 비대화형 환경 → 컨텍스트 자동 감지만으로 Phase 1 진행
```

**비대화형 환경 판단 기준:** TTY 없음 (`[ -t 0 ]` false), 또는 `CI=true` / `GITHUB_ACTIONS` / `GITLAB_CI` 환경변수 존재 시

**비대화형 환경 자동 감지 항목:**
- `package.json` → 테스트 스크립트, 프레임워크
- `pytest.ini` / `pyproject.toml` → Python 테스트 설정
- `playwright.config.*` / `cypress.config.*` → E2E 설정
- `.env` / `.env.local` → API URL 추정
- `git remote` → GitHub / GitLab 구분

### Phase 1: 컨텍스트 분석

- PR 번호 감지: `git log`, `gh pr view`, GitLab MR API
- PR 있음 → 변경 파일 diff 분석 → 기능 분류
- PR 없음 → 사용자에게 테스트할 기능 질문 (비대화형이면 전체 스캔)

**변경 분류:**
| 분류 | 감지 기준 | 실행할 테스트 |
|------|-----------|--------------|
| API 변경 | `routes/`, `controllers/`, `*.api.*` 파일 | 단위 + API 호출 |
| UI 변경 | `components/`, `pages/`, `*.tsx`, `*.vue` | 단위 + E2E |
| 로직 변경 | `services/`, `utils/`, `lib/` | 단위 + 통합 |
| 설정 변경 | `*.config.*`, `.env*` | 연기 테스트 (환경 의존) |
| 복합 변경 | 위 2가지 이상 | 전체 조합 |

### Phase 2: 테스트 계획 수립

- 분류 결과 기반으로 테스트 전략 결정
- `test_depth` 기준으로 TC 수 조절:
  - `minimal`: 주요 GREEN CASE만
  - `standard`: GREEN + 주요 EDGE CASE (기본값)
  - `thorough`: GREEN + 전체 EDGE CASE + 경계값
- `plan-report.html` 생성

### Phase 3: 테스트 실행

실행 우선순위: 단위 → 통합 → API → E2E

**단위/통합:** SUPERTEST.md의 `unit` 명령 또는 자동 감지 명령 실행  
**API:** `curl` / `fetch`로 엔드포인트 호출, 인증 설정 적용  
**E2E:** Playwright / Cypress 실행 (설정 있을 때만)

실패 시: 에러 로그 캡처 + 원인 추정 (`error_cause` 필드에 기록)

### Phase 4: 보고서 생성

- `result-report.html` 생성
- `executive-summary.html` 생성
- PR 컨텍스트 + `pr_comment.enabled: true` → `gh pr comment` 또는 GitLab API로 게시

## SUPERTEST.md 구조

```markdown
# SUPERTEST Configuration

## Project
- name: my-app
- type: fullstack  # api-only | frontend-only | fullstack | mobile

## Test Scope
- unit: true
- integration: true
- e2e: true
- api: true

## Test Commands
- unit: npm test
- e2e: npx playwright test
- coverage: npm run test:coverage

## API Testing
- base_url: http://localhost:3000
- auth_method: bearer        # bearer | basic | cookie | none
- auth_token_env: API_TOKEN  # 환경변수명 (값은 저장하지 않음)
- auth_cookie_name: session

## Browser Testing
- browser: chromium          # chromium | firefox | webkit
- headless: true
- base_url: http://localhost:3000

## Reports
- output_dir: .supertest     # 기본값

## PR Comment
- enabled: true
- post_on: github            # github | gitlab | none
- format: summary            # summary | detailed | minimal
- include_tc_table: true
- include_failed_logs: true
- include_coverage: false

## PR Comment Template
- header: "## 🧪 SuperTest Report"
- pass_icon: "✅"
- fail_icon: "❌"
- skip_icon: "⏭️"
- footer: "Generated by supertest skill"

## Test Depth
- level: standard            # minimal | standard | thorough

## Exclusions
- paths:
    - node_modules
    - dist
    - .next
    - coverage
```

**init 인터뷰 — Step 0: 모노리포 / 멀티 언어 감지**

init 인터뷰 시작 전, 아래 파일을 탐색해 프로젝트 구조를 자동 감지한다:

| 감지 파일 | 의미 |
|-----------|------|
| `pnpm-workspace.yaml`, `nx.json`, `turbo.json`, `lerna.json` | 모노리포 |
| 루트 외 여러 `package.json` | JS 멀티 패키지 |
| 서로 다른 디렉토리에 `pyproject.toml` + `package.json` | 멀티 언어 |
| `go.mod`, `Cargo.toml`, `build.gradle` 혼재 | 멀티 언어 |

모노리포 또는 멀티 언어로 감지되면, 각 워크스페이스/디렉토리를 분석하여 아래 **확인 테이블**을 사용자에게 제시하고 컨펌을 받는다:

```
감지된 프로젝트 구조를 확인해주세요:

| # | 경로              | 타입        | 테스트 프레임워크 | E2E        | API 테스트 | 실행 명령          |
|---|-------------------|-------------|-------------------|------------|------------|--------------------|
| 1 | packages/frontend | frontend    | Vitest            | Playwright | -          | pnpm test          |
| 2 | packages/backend  | api-only    | Jest              | -          | curl       | pnpm test          |
| 3 | services/auth     | api-only    | Pytest            | -          | curl       | pytest             |
| 4 | apps/mobile       | mobile      | Jest              | Detox      | -          | yarn test          |

수정할 항목이 있으면 번호와 변경 내용을 알려주세요. 없으면 '확인'을 입력하세요.
```

컨펌 후, SUPERTEST.md에 워크스페이스별 설정을 저장한다 (아래 참고).

단일 프로젝트로 감지되면 기존 인터뷰 흐름을 그대로 사용한다.

**init 인터뷰 질문 목록 (단일 프로젝트 또는 워크스페이스별 공통 항목):**
1. 프로젝트 타입 (api-only / frontend-only / fullstack / mobile)
2. 사용 중인 테스트 프레임워크 및 실행 명령
3. E2E 테스트 도구 (Playwright / Cypress / 없음)
4. API 베이스 URL
5. 인증 방식 (bearer / basic / cookie / none) + 환경변수명
6. 브라우저 테스트 설정 (브라우저 종류, headless 여부)
7. 테스트 깊이 (minimal / standard / thorough)
8. PR 코멘트 자동 게시 여부 + 플랫폼 (GitHub / GitLab)
9. PR 코멘트 상세도 (summary / detailed / minimal)
10. 제외할 경로

**모노리포일 때 SUPERTEST.md 추가 섹션:**

```markdown
## Workspaces
- monorepo: true

### packages/frontend
- type: frontend
- unit_command: pnpm test
- e2e_command: pnpm playwright test
- base_url: http://localhost:5173

### packages/backend
- type: api-only
- unit_command: pnpm test
- api_base_url: http://localhost:3000
- auth_method: bearer
- auth_token_env: API_TOKEN

### services/auth
- type: api-only
- unit_command: pytest
- api_base_url: http://localhost:8000
- auth_method: bearer
- auth_token_env: AUTH_TOKEN
```

## TC 구조

```
TC-001 | 기능명
  type:     GREEN | EDGE
  input:    { endpoint, payload, 화면 액션 등 }
  expected: HTTP 200 / 요소 존재 / 값 일치 / 에러 메시지 포함
  method:   unit | api | e2e
  status:   PENDING → PASS | FAIL | SKIP
  error:    (FAIL 시) 에러 메시지 원문 + 원인 추정
```

## HTML 보고서 3종

### plan-report.html
- 테스트 대상 기능 목록
- TC 전체 목록 (GREEN/EDGE 구분, 실행 방법)
- 테스트 전략 요약 (프레임워크, 실행 순서)
- 예상 소요 시간

### result-report.html
- TC별 PASS / FAIL / SKIP 상태 테이블
- FAIL 항목: 에러 로그 + 원인 추정 섹션
- 전체 통계 (통과율, 실행 시간)
- 스택트레이스 접이식(collapsible) 표시

### executive-summary.html
- 총 TC / PASS / FAIL / SKIP 수치
- 위험도 평가 (핵심 기능 FAIL → 🔴, 엣지케이스 FAIL → 🟡, 전체 PASS → 🟢)
- 한 줄 결론: `배포 가능` / `조건부 배포 가능` / `배포 보류 권장`
- 테스트 범위 및 제외 항목

### PR 코멘트 기본 형식
```markdown
## 🧪 SuperTest Report

| 항목 | 결과 |
|------|------|
| 총 TC | 24 |
| ✅ PASS | 21 |
| ❌ FAIL | 2 |
| ⏭️ SKIP | 1 |

**위험도:** 🟡 조건부 배포 가능

<details>
<summary>실패 TC 보기</summary>

| TC | 기능 | 에러 | 원인 추정 |
|----|------|------|-----------|
| TC-007 | 로그인 API | 401 Unauthorized | 토큰 만료 또는 잘못된 시크릿 |

</details>

Generated by supertest skill
```

## 파일 구조

```
supertest/
├── .claude-plugin/
│   └── plugin.json
└── skills/
    └── supertest/
        ├── SKILL.md             # 전체 플로우 제어, Phase 분기 로직
        ├── init.md              # init 인터뷰 지침 + SUPERTEST.md 생성 템플릿
        ├── tc-strategy.md       # TC 작성 전략 (GREEN/EDGE 기준, 분류별 패턴)
        ├── html-templates.md    # 3종 HTML 보고서 템플릿
        └── pr-comment.md        # PR 코멘트 생성 지침 + 기본 템플릿
```

| 파일 | 로드 시점 |
|------|-----------|
| `SKILL.md` | 항상 |
| `init.md` | Phase 0, SUPERTEST.md 없을 때 |
| `tc-strategy.md` | Phase 2 |
| `html-templates.md` | Phase 2, 4 |
| `pr-comment.md` | Phase 4, PR 있을 때 |


## 제약 및 경계

- 코드 리뷰 기능 없음 (테스트 실행 및 보고에만 집중)
- 에러 원인 추정은 보고서에 포함되지만 수정 제안은 하지 않음
- API 인증 토큰 값은 SUPERTEST.md에 저장하지 않음 (환경변수명만 저장)
- E2E 테스트는 playwright/cypress 설정 파일이 감지될 때만 실행
- 비대화형 환경에서는 인터뷰 없이 자동 감지 결과로만 동작
